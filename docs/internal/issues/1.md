# 当前最新状态判断

## 1. Compiler physical DOM path 已经接进 pipeline

现在 `transformJSX()` 已经按这个顺序执行：

```ts
normalizeChildren(ir)
validateBuiltins(ir)
assignDomPaths(ir)
assignPhysicalDomPaths(ir)
analyzeBindings(ir)
collectTemplates(ir, context)
```

说明 physical DOM path 方案已经正式进入编译主流程了。

`nodes.ts` 里也已经有：

```ts
export type PhysicalDomPath =
  | { kind: 'Root' }
  | { kind: 'FirstChild'; parent: IRRef }
  | { kind: 'NextSibling'; previous: IRRef }
  | { kind: 'ChildNode'; parent: IRRef; index: number }
```

并且 `ElementIR / DynamicTextIR / ComponentIR / ShowIR / ForIR / SlotIR` 都已经带了 `physicalDomPath`。

这部分方向是对的。

---

## 2. emitElement 已经改成 bind 前提前声明 DOM refs

你现在 `emitElement()` 已经是：

```ts
const _el$ = emitTemplateClone(...)
...emitDomRefDeclarations(node.children, context)
...emitBindings(node, context)
return _el$
```

这正好符合 Solid 风格：**先拿 DOM/anchor 引用，再 insert/bind/mount**。

但是当前还有一个必须修的坑：`emitDomRefDeclarations()` 会跳过无 runtime work 的静态元素，而 `assignPhysicalDomPaths()` 的 `NextSibling` 可能依赖这些被跳过的静态元素。你前面遇到的 `_el$2 is not defined` 就是这个问题。

---

## 3. DynamicText 已经基本按 anchor 语义改了

现在 `emitDynamicText()` 已经不再把 `node.ref` 当 Text 节点，而是创建临时 `textRef`，然后：

```ts
_insert(parent, textRef, node.ref)
_bindText(textRef, () => expr)
```

这说明 `DynamicTextIR.ref` 已经实际变成了 anchor ref。

但 `lowerExpression()` 里还在用：

```ts
ref(context.uid('text$').name)
```

这个只是命名问题，不影响运行，但建议改成 `anchor$`，否则后续自己看代码会混淆。

---

## 4. runtime-dom 已经进入 Phase 6 能力

runtime-dom 现在已经导出了 `defineElement / Host / Slot / createSlot / hostContext`，说明 Web Components 能力已经落地到入口层。

事件系统已经是委托模式：

```ts
bindEvent() 只把 handler 存到 el.__zeusEvents
delegateEvents() 在 document 上注册一次事件
```

这已经不是 MVP 的直接 `addEventListener` 了。

`mountFor()` 也已经支持 `key` 函数，能做 keyed diff / DOM move。

# 第一优先级：修 compiler physical path 的依赖闭包

这是当前最紧急的。

## 问题

现在 `assignPhysicalDomPaths()` 会生成：

```ts
const _el$3 = _el$2.nextSibling
```

但 `emitElement()` 可能不会声明 `_el$2`，因为 `_el$2` 对应的是无动态绑定的静态元素。

## 修复目标

声明某个节点前，递归声明它的依赖：

```txt
NextSibling(previous) -> previous 必须先声明
FirstChild(parent)   -> parent 必须已声明
ChildNode(parent)    -> parent 必须已声明
```

## 补丁草案

替换 `emitElement.ts` 里的 DOM ref declaration 逻辑。

```ts
function emitDomRefDeclarations(
  children: ZeusIRNode[],
  context: CompilerContext,
): t.Statement[] {
  const statements: t.Statement[] = []
  const declared = new Set<string>()
  const refNodeMap = collectRefNodeMap(children)

  for (const child of children) {
    collectRequiredDomRefDeclarations(
      child,
      statements,
      context,
      refNodeMap,
      declared,
    )
  }

  return statements
}

function collectRefNodeMap(children: ZeusIRNode[]): Map<string, DomRefNode> {
  const map = new Map<string, DomRefNode>()

  for (const child of children) {
    collectRefNode(child, map)
  }

  return map
}

function collectRefNode(node: ZeusIRNode, map: Map<string, DomRefNode>): void {
  switch (node.kind) {
    case 'Element':
      map.set(node.ref.name, node)

      for (const child of node.children) {
        collectRefNode(child, map)
      }

      return

    case 'DynamicText':
    case 'Component':
    case 'Show':
    case 'For':
    case 'Slot':
      map.set(node.ref.name, node)
      return

    case 'Fragment':
    case 'Host':
      for (const child of node.children) {
        collectRefNode(child, map)
      }
      return

    default:
      return
  }
}
```

然后声明时补依赖：

```ts
function collectRequiredDomRefDeclarations(
  node: ZeusIRNode,
  statements: t.Statement[],
  context: CompilerContext,
  refNodeMap: Map<string, DomRefNode>,
  declared: Set<string>,
): void {
  switch (node.kind) {
    case 'Element':
      if (needsDomRefDeclaration(node)) {
        emitDomRefDeclarationWithDeps(
          node,
          statements,
          context,
          refNodeMap,
          declared,
        )
      }

      for (const child of node.children) {
        collectRequiredDomRefDeclarations(
          child,
          statements,
          context,
          refNodeMap,
          declared,
        )
      }

      return

    case 'DynamicText':
    case 'Component':
    case 'Show':
    case 'For':
    case 'Slot':
      emitDomRefDeclarationWithDeps(
        node,
        statements,
        context,
        refNodeMap,
        declared,
      )
      return

    case 'Fragment':
    case 'Host':
      for (const child of node.children) {
        collectRequiredDomRefDeclarations(
          child,
          statements,
          context,
          refNodeMap,
          declared,
        )
      }
      return

    default:
      return
  }
}

function emitDomRefDeclarationWithDeps(
  node: DomRefNode,
  statements: t.Statement[],
  context: CompilerContext,
  refNodeMap: Map<string, DomRefNode>,
  declared: Set<string>,
): void {
  if (declared.has(node.ref.name)) return

  if (!node.physicalDomPath) {
    throw new Error(`${node.kind} physical DOM path is not assigned`)
  }

  emitPhysicalDomPathDependencies(
    node.physicalDomPath,
    statements,
    context,
    refNodeMap,
    declared,
  )

  statements.push(createDomRefDeclaration(node, context))
  declared.add(node.ref.name)
}

function emitPhysicalDomPathDependencies(
  path: PhysicalDomPath,
  statements: t.Statement[],
  context: CompilerContext,
  refNodeMap: Map<string, DomRefNode>,
  declared: Set<string>,
): void {
  switch (path.kind) {
    case 'Root':
      return

    case 'FirstChild':
    case 'ChildNode':
      emitRefDependency(path.parent, statements, context, refNodeMap, declared)
      return

    case 'NextSibling':
      emitRefDependency(
        path.previous,
        statements,
        context,
        refNodeMap,
        declared,
      )
      return
  }
}

function emitRefDependency(
  ref: IRRef,
  statements: t.Statement[],
  context: CompilerContext,
  refNodeMap: Map<string, DomRefNode>,
  declared: Set<string>,
): void {
  const dep = refNodeMap.get(ref.name)

  // 找不到通常代表当前 IIFE root，比如 _el$，它已经提前声明。
  if (!dep) return

  emitDomRefDeclarationWithDeps(dep, statements, context, refNodeMap, declared)
}
```

修完后 counter 应该从错误产物：

```ts
const _el$3 = _el$2.nextSibling
```

变成：

```ts
const _el$2 = _el$.firstChild
const _el$3 = _el$2.nextSibling
```

# 第三优先级：修 Vite plugin 的 alias 时机问题

你现在 Vite plugin 里：

```ts
let viteRoot = ''

configResolved(config) {
  viteRoot = config.root
}

config() {
  return {
    resolve: {
      alias: {
        '@zeus-js/runtime-dom': `${viteRoot}/node_modules/@zeus-js/runtime-dom`,
        '@zeus-js/zeus': `${viteRoot}/node_modules/@zeus-js/zeus`,
      },
    },
  }
}
```

问题是 `config()` 执行时 `configResolved()` 还没跑，所以 `viteRoot` 还是空字符串。

## 建议修法

短期最简单：**先删除这两个 alias**，改用 `resolve.dedupe`。

```ts
config() {
  return {
    esbuild: {
      jsx: 'preserve',
    },
    oxc: {
      jsx: 'preserve',
    },
    resolve: {
      dedupe: [
        '@zeus-js/runtime-dom',
        '@zeus-js/zeus',
        '@zeus-js/signal',
      ],
    },
  }
}
```

你现在生成的 import 里已经出现了 `/@fs/.../dist/runtime-dom.esm-bundler.js`，说明 Vite 正在处理 workspace/dist 路径。alias 不要在这里强行拼 root，容易造成 Windows/macOS 路径和 pnpm workspace 问题。

# 第四优先级：统一事件委托开关

现在 Vite plugin 传给 compiler 的配置里是：

```ts
delegateEvents: false
```

但 compiler 的 `Program.exit` 不管配置，始终执行：

```ts
appendTemplates(path)
appendEvents(path)
appendImportMethods(path)
```

同时 `emitEventBinding()` 每次都会 `registerEvent(...)`。

而 runtime 的 `bindEvent()` 现在又是委托模式，只存 handler，不直接 `addEventListener`。

所以现在事实上是：

```txt
delegateEvents 必须开启，否则事件不会工作。
```

## 短期建议

把 Vite plugin 里的：

```ts
delegateEvents: false
```

改成：

```ts
delegateEvents: true
```

或者直接不传，让 compiler 默认值生效。compiler 默认 `delegateEvents` 本来就是 `true`。

## 后续再做

如果你以后真想支持非委托事件，需要拆两个 runtime helper：

```ts
bindEvent() // 直接 addEventListener
bindDelegatedEvent() // 存 __zeusEvents
```

但现在没必要。

# 第五优先级：修 For keyed diff 的语义边界

你现在 `mountFor()` 已经支持 key 函数。

但是有一个隐性问题：

```ts
const list = state([{ id: 1, title: 'a' }])

list[0] = { id: 1, title: 'b' }
```

当前 `mountKeyedFor()` 如果 key 相同，会复用 oldRecord.nodes，但 `render(item, i)` 不会重新执行，旧 DOM 里的闭包可能还指向旧 item。当前代码只是：

```ts
oldRecord.item = item
oldRecord.index = i
nextRecords.push(oldRecord)
```

并不会让 DOM 中的表达式换成新 item。

## 短期规则

先文档化：

```txt
For keyed 模式下，推荐 item 本身是 reactive object；
替换同 key 的普通对象不保证局部 DOM 更新。
```

## 中期修法

给每个 record 一个稳定的 `itemState`：

```ts
type ListRecord<T> = {
  key: Key
  item: ValueState<T>
  index: ValueState<number>
  nodes: Node[]
}
```

render 时传：

```ts
render(record.item.value, record.index.value)
```

但这样 JSX 里 `todo.title` 仍然拿的是当次 value，不是 getter。更彻底的做法是给 For item 做编译级特殊处理，这个可以后面做，不要现在卡住。

当前阶段先补测试，明确边界。
