## 必须修复 1：`addons/vite-plugin` 的 d.ts 默认不会生成

这是当前最大问题。

`tsconfig.build.json` 已经包含了：

```txt
addons/vite-plugin/src
```

说明 tsc 会为 vite-plugin 生成临时声明文件。

但是 `scripts/bundler/rolldown.dts.config.ts` 默认只读取：

```ts
const tempPkgs = readdirSync('temp/packages')
```

然后默认目标来自 `tempPkgs`。

这意味着默认执行：

```bash
pnpm build-dts
```

大概率只会给 `packages/*` 生成 d.ts，不会给 `addons/vite-plugin` 生成：

```txt
addons/vite-plugin/dist/vite-plugin.d.ts
```

但 `addons/vite-plugin/package.json` 的 exports 明确声明了这个文件：

```json
"types": "./dist/vite-plugin.d.ts"
```

而 `check-package-exports.ts` 会检查所有非 private workspace 包的 exports 文件是否存在。

所以 `release-precheck` 里跑到 `check:exports` 时，有可能直接失败。

建议改成：

```ts
const tempRoots = ['temp/packages', 'temp/addons']

const discoveredTargets = findWorkspacePackages().filter(pkg => {
  const category = pkg.relativeDir.split('/')[0]
  return existsSync(`temp/${category}/${pkg.shortName}/src/index.d.ts`)
})
```

或者最小修复：

```ts
const tempPkgs = [
  ...safeReadDir('temp/packages'),
  ...safeReadDir('temp/addons'),
]
```

但更推荐基于 `findWorkspacePackages()` 来推导，避免后续新增 `devtools/*` 又漏掉。

## 必须修复 2：`tsconfig.paths` 没有覆盖 `addons/*`

现在 `tsconfig.json` 是：

```json
"@zeus-js/*": ["./packages/*/src"]
```

但 include 里已经包含了 `addons/*/src`。

这会导致一个问题：如果测试、examples 或内部源码直接 import：

```ts
import zeus from '@zeus-js/vite-plugin'
```

TypeScript 路径映射找不到 `addons/vite-plugin/src`。

虽然当前 `addons/vite-plugin` 自己主要 import `@zeus-js/compiler`，暂时不一定触发，但既然 `aliases.ts` 运行时 alias 已经扫描 `packages` 和 `addons`，TS paths 也应该一致。

建议改成：

```json
"paths": {
  "*": ["./*"],
  "@zeus-js/*": [
    "./packages/*/src",
    "./addons/*/src"
  ]
}
```

## 必须修复 3：`addons/vite-plugin` 的 npm 元信息还是旧路径

`addons/vite-plugin/package.json` 已经移动到 `addons/vite-plugin`，但里面还是：

## 建议修复 5：`findWorkspacePackages()` 不应该吞异常

现在这里：

```ts
try {
  const packageJson = require(pkgJsonPath)
  results.push(...)
} catch {}
```

如果某个 package.json 写坏了，它会被静默跳过。对于 build/release 脚本来说，这很危险。

建议改成：

```ts
} catch (error) {
  throw new Error(`Failed to read workspace package: ${pkgJsonPath}`, {
    cause: error,
  })
}
```

## 建议修复 6：`aliases.ts` 最好复用 `findWorkspacePackages()`

现在 `aliases.ts` 自己重新扫描 `packages` 和 `addons`。

这不是大问题，但已经出现重复规则了：

```txt
utils.ts     扫 packages/addons
aliases.ts   也扫 packages/addons
rolldown     也写了 packages/addons
```

后面如果你新增 `devtools/*`，很容易漏改。更好的方式是：

```ts
import { findWorkspacePackages } from './utils'

export const entries = Object.fromEntries(
  findWorkspacePackages().map(pkg => [
    pkg.name,
    path.resolve(pkg.dir, 'src/index.ts'),
  ]),
)
```

注意这里应该用 `pkg.name`，不是固定拼 `@zeus-js/${dir}`。因为 `create-zeus` 这种包名不是 `@zeus-js/create-zeus`。

## 建议修复 7：banner 和 `__VERSION__` 仍使用 root version

`rolldown.config.ts` 现在读取 root package version：

```ts
const masterVersion = require('../../package.json').version
```

并用于 banner 和 `__VERSION__`。

这对 core fixed 包暂时可以，但对 `addons/vite-plugin` 这种独立版本包不准确。建议改成：

```ts
const pkg = require(resolve('package.json'))
const packageVersion = pkg.version
```

然后：

```ts
__VERSION__: JSON.stringify(packageVersion)
```

这样 vite-plugin 独立发版时不会显示 root/core 版本。

## 发版流程评价

`release-precheck` 会先执行 `build`、`build-dts`、`check`、`lint`、`test-unit`、examples、docs、size 和 exports 检查，这个流程是好的。

Release workflow 里 `Build and publish` 仍然是：

```yaml
pnpm release --publishOnly ${{ env.VERSION }} --skipBuild
```

因为前一步 `release:precheck` 已经 build 过，所以这个不一定会断。但我建议至少加注释，说明 `--skipBuild` 依赖 precheck 产物；否则以后有人删掉 precheck 里的 build，会导致发布空 dist。
