结论：**不建议现在直接合并到 main**。
可以认为 **“技术 MVP 基本完成”**，但 **“可合并主线 / 可发布 MVP” 还差几个阻断项**。我的建议是：先修完下面的 P0，再合并；否则 main 会带着几个明显的包出口和示例可用性问题。

# 当前不建议合并的 P0 阻断项

## P0-1：`@zeus-js/zeus/advanced` 和 `@zeus-js/zeus/internal` 的 package exports 很可能是错的

你已经有：

```txt
packages/zeus/src/advanced.ts
packages/zeus/src/internal.ts
```

`advanced.ts` 导出了 `stop / effectScope / getCurrentScope / queueJob / ReactiveFlags` 等高级 API。
`internal.ts` 导出了 `template / insert / bindText / bindEvent / mountShow / mountFor / createSlot` 等内部 runtime helper。

但 `packages/zeus/package.json` 里：

```json
"./advanced": {
  "types": "./dist/zeus.d.ts",
  "import": "./dist/zeus.esm-bundler.js",
  "require": "./dist/zeus.cjs.js"
},
"./internal": {
  "types": "./dist/zeus.d.ts",
  "import": "./dist/zeus.esm-bundler.js",
  "require": "./dist/zeus.cjs.js"
}
```

也就是说，`@zeus-js/zeus/advanced` 和 `@zeus-js/zeus/internal` 实际都指向主 bundle `dist/zeus.*`，而主入口 `src/index.ts` 并没有导出这些 advanced/internal API。

这会导致：

```ts
import { stop } from '@zeus-js/zeus/advanced'
```

在类型或运行时层面都可能不可用。

**修复建议：**

短期二选一：

```txt
方案 A：暂时删除 ./advanced 和 ./internal exports，等多入口构建好了再加。
方案 B：改 build 系统支持 zeus 多入口，产出 dist/advanced.* 和 dist/internal.*。
```

如果要保留当前设计，应该改成：

```json
"./advanced": {
  "types": "./dist/advanced.d.ts",
  "import": "./dist/advanced.esm-bundler.js",
  "require": "./advanced.js"
},
"./internal": {
  "types": "./dist/internal.d.ts",
  "import": "./dist/internal.esm-bundler.js",
  "require": "./internal.js"
}
```

并确保构建确实产出这些文件。

## P0-2：`check-package-exports.ts` 会放过 `.d.ts` 缺失，掩盖了上面的问题

当前 exports 检查脚本明确跳过了所有 `.d.ts` 文件：

```ts
if (target.endsWith('.d.ts')) return
```

这意味着即使 `./advanced` 指向不存在或不正确的 types，它也不会报错。对于要发布 npm 包来说，这个检查不够严格。

**修复建议：**

至少不要跳过 `.d.ts`：

```ts
if (target.endsWith('.d.ts')) return
```

这行应该删掉。否则 `release:precheck` 不能证明包出口真的可用。

---

## P0-3：`state(Map/Set)` 类型和运行时仍然不一致

`state.ts` 运行时会把 `Map / Set / WeakMap / WeakSet` 当作 proxyable：

```ts
value instanceof Map ||
  value instanceof Set ||
  value instanceof WeakMap ||
  value instanceof WeakSet
```

但类型层面的 `ProxyableInput` 仍然只有：

```ts
type ProxyableInput = Record<PropertyKey, any> | readonly any[]
```

这会导致：

```ts
const map = state(new Map<string, number>())
```

运行时是 reactive Map，但 TS 类型可能变成 `ValueState<Map<...>>`。这是 API 级不一致，建议合并前修掉。

**修复建议：**

```ts
type ProxyableInput =
  | Record<PropertyKey, any>
  | readonly any[]
  | Map<unknown, unknown>
  | Set<unknown>
  | WeakMap<object, unknown>
  | WeakSet<object>
```

## P0-4：`examples/todo` 使用了 `value={...}` / `checked={...}`，但 compiler 只有 `prop:` 才会走 DOM property

`lowerAttribute.ts` 只有 `prop:` 前缀会生成 `PropBindingIR`：

```ts
if (name.startsWith('prop:')) {
  return propBindingIR(name.slice('prop:'.length), expr)
}
return attrBindingIR(name, expr)
```

而 `bindAttr()` 只是 `setAttribute/removeAttribute`，不会设置 `HTMLInputElement.value` 或 `HTMLInputElement.checked` property。

但 `examples/todo/src/main.tsx` 里用了：

```tsx
<input value={input.value} ... />
<input checked={todo.done} ... />
```

这会让 Todo 示例出现“看似编译通过，但表单受控行为不正确”的问题，特别是 `input.value = ''` 后 UI 不一定被 property 清空。

**修复建议：**

把示例改成：

```tsx
<input
  prop:value={input.value}
  onInput={e => {
    input.value = (e.currentTarget as HTMLInputElement).value
  }}
/>

<input
  type="checkbox"
  prop:checked={todo.done}
  onChange={e => {
    todo.done = (e.currentTarget as HTMLInputElement).checked
  }}
/>
```

或者 compiler 层内置规则：`value/checked/selected` 默认走 property，但这会引入语义决策，MVP 阶段更建议先显式用 `prop:`。

---

## P0-5：`create-zeus` 目前不会被 release precheck 构建，也被 changeset ignore 了

`create-zeus` 的 `bin` 指向 `./dist/index.js`，它自己的 build 脚本是 `tsup src/index.ts ...`。

但根 `pnpm build` 只会构建 `packages/*` 下带 `buildOptions` 且非 private 的包；`create-zeus` 没有 `buildOptions`，所以不会被根 build 构建。

同时 `.changeset/config.json` 把 `create-zeus` ignore 了。

所以如果你想把 `create-zeus` 作为 MVP 一部分发布，现在是不完整的。
如果不准备发布 create-zeus，那问题不大，但文档里不要主打 `pnpm create zeus`。

**修复建议：**

短期：

```json
"release:precheck": "tsx scripts/release-precheck.ts"
```

里面加一步：

```ts
;['pnpm', ['create-zeus:build']]
```

或者暂时明确：

```txt
v0.1.0-alpha 不发布 create-zeus
```

# P1 风险项

## P1-1：Vite plugin 只配置了 `oxc.jsx = preserve`，没有 `esbuild.jsx = preserve`

当前 Vite plugin：

```ts
config() {
  return {
    oxc: {
      jsx: 'preserve',
    },
    resolve: {
      dedupe: [...]
    },
  }
}
```

因为插件 `enforce: 'pre'`，多数情况下能先拿到 TSX 并编译掉 JSX。
但为了兼容 Vite/esbuild 路径，建议加回：

```ts
esbuild: {
  jsx: 'preserve',
}
```

这不是绝对阻断项，但属于 DX 稳定性修复。

---

## P1-2：`For` 同 key 替换普通对象仍有语义边界

`mountKeyedFor()` 在 key 相同时复用 old record，只更新 `oldRecord.item` / `oldRecord.index`，不会重新 render 子树。

这意味着同 key 普通对象替换不是完整重渲染。这个可以接受，但必须文档明确。当前 docs 里已经有 runtime semantics 和 control-flow 文档，但合并前建议确保这条限制写清楚。compare 里已有 `docs/guide/runtime-semantics.md` 和 `docs/guide/control-flow.md`，说明文档方向已经具备。
