结论：**MVP 代码可以认为已经过线；除版本号问题外，没有看到必须阻塞合并/发布 alpha 的 P0。**

我这次看的是仓库默认分支 `main`，不是旧的 `feat/signal` 分支。当前主线已经具备 MVP 闭环：`state → TSX compiler → runtime-dom → Vite plugin → examples → docs → release precheck`。默认分支是 `main`，而主线根脚本已经包含 `build / build-dts / check / lint / test-unit / docs:build / examples:check:all / release:precheck / release:dry` 等发布前闸门。

# 代码层面做得好的点

## 1. Compiler 物理 DOM path 方案已经比较稳

`transformJSX()` 的流程是：

# 我看到的非阻塞问题

## P1：`defineElement.connectedCallback()` 里有一段 `onScopeDispose()` 基本是无效/误导的

在 `connectedCallback()` 末尾有：

```ts
onScopeDispose(() => {
  this.dispose?.()
  this.dispose = undefined
}, true)
```

但它是在 `render()` 调用完成之后执行的，已经不在 `renderScope.run()` 的 active scope 里；并且传了 `true`，没有 active scope 时不会警告，所以这段大概率不会注册到任何 scope 上。真正生效的是 `disconnectedCallback()` 里的 `this.dispose?.()`。

建议改法：直接删掉 `connectedCallback()` 里的 `onScopeDispose` 那段，避免让维护者误以为它参与生命周期管理。

这不是 P0，因为 `disconnectedCallback()` 已经兜住了。

## P2：`mountKeyedFor()` 的 devtools 事件 length 用的是旧 records 长度

当前逻辑里：

```ts
emitDevtoolsEvent({ type: 'mount-for', length: records.length })
records = nextRecords
```

也就是上报的是更新前的长度，不是本次渲染后的长度。

建议改成：

```ts
emitDevtoolsEvent({ type: 'mount-for', length: nextRecords.length })
records = nextRecords
```

这只影响 devtools/debug 数据，不影响 DOM 行为。

## P2：`event.currentTarget` 的重写需要浏览器 e2e 再兜一层

现在实现是通过 `Object.defineProperty(event, 'currentTarget', ...)` 临时覆盖，如果环境不允许覆盖，会进入 catch，此时只保证 `handler.call(el, event)` 的 `this` 正确，但箭头函数里读 `event.currentTarget` 可能不正确。代码里也有这个 fallback 注释。

这个在 jsdom/unit 里通常可以过，但建议后续加一个真实浏览器 e2e，验证 Chrome/Safari/Firefox 里 `event.currentTarget` 都符合预期。
