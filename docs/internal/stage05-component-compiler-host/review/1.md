# P0：必须修复后再合并

## 1. Vue wrapper 生成代码会引用未定义变量

`generateVuePropSyncLines()` 当前生成的是：

```ts
if (variant !== undefined) el.variant = variant
else el.variant = undefined
```

但在 Vue wrapper 的 `setup(props, ...)` 中并没有解构出 `variant` / `disabled` 这些变量，实际应该是：

```ts
if (props.variant !== undefined) el.variant = props.variant
else el.variant = undefined
```

当前实现里生成逻辑确实使用了裸变量名：`variant !== undefined`、`el.variant = variant`。测试里甚至也在断言这个错误字符串，说明测试没有覆盖“生成代码能真实运行”。

建议修成：

```ts
function generateVuePropSyncLines(propNames: string[]): string {
  if (propNames.length === 0) {
    return '// no props to sync'
  }

  return propNames
    .map(
      name =>
        `if (props.${name} !== undefined) el.${name} = props.${name}; else el.${name} = undefined;`,
    )
    .join('\n      ')
}
```

同时把测试改成断言：

```ts
expect(code).toContain('el.variant = props.variant')
expect(code).not.toContain('el.variant = variant')
```

## 2. `@zeus-js/bundler-plugin/vite` 在 ESM 环境下直接使用 `require.resolve`

`addons/bundler-plugin/src/vite.ts` 是 ESM 源文件，但 `resolveRuntimeDOMEntry()` 里直接调用了 `require.resolve`。在 ESM 运行时，`require` 不一定存在，这会导致 Vite 插件在某些环境直接炸。

建议改成：

```ts
import { createRequire } from 'node:module'

const require = createRequire(import.meta.url)
```

或者更稳：

```ts
function resolveRuntimeDOMEntry(root: string | undefined): string | undefined {
  const projectRoot = path.resolve(process.cwd(), root ?? '.')
  const requireFromProject = createRequire(
    path.join(projectRoot, 'package.json'),
  )

  try {
    return requireFromProject.resolve(
      '@zeus-js/runtime-dom/dist/runtime-dom.esm-bundler.js',
    )
  } catch {}

  try {
    const zeusEntry = requireFromProject.resolve('@zeus-js/zeus')
    const requireFromZeus = createRequire(zeusEntry)

    return requireFromZeus.resolve(
      '@zeus-js/runtime-dom/dist/runtime-dom.esm-bundler.js',
    )
  } catch {
    return undefined
  }
}
```

## 3. `wc/index.d.ts` 里的 `HTMLElementTagNameMap` 可能引用不到本地类型

`generateWCIndexDts()` 当前先 `export * from "./z-button"`，然后在 `declare global` 里直接使用 `ZButtonElement`。但 `export * from` 不保证把 `ZButtonElement` 引入当前模块本地作用域，`HTMLElementTagNameMap` 里直接写 `ZButtonElement` 有概率类型不可解析。

当前生成大概是：

```ts
export * from './z-button'

declare global {
  interface HTMLElementTagNameMap {
    'z-button': ZButtonElement
  }
}
```

建议生成：

```ts
import type { ZButtonElement } from './z-button'
export * from './z-button'

declare global {
  interface HTMLElementTagNameMap {
    'z-button': ZButtonElement
  }
}
```

也就是 `generateWCIndexDts()` 对每个 component 先生成 `import type`，再 `export *`。

# P1：建议合并前修复

## 4. `output-wc` 默认 `outDir` 是 `dist/wc`，容易在 Vite 里输出成 `dist/dist/wc`

`output-wc` 当前默认：

```ts
outDir: options.outDir ?? 'dist/wc'
manifestFile: options.manifestFile ?? 'dist/zeus.components.json'
customElementsFile: options.customElementsFile ?? 'dist/custom-elements.json'
```

但 Rollup/Vite 的 `emitFile.fileName` 是相对于最终 output dir 的路径，所以在 Vite 默认 `build.outDir = dist` 时，这个默认值会得到 `dist/dist/wc`。你在 `packages/headless/vite.config.ts` 里已经手动传了 `outDir: 'wc'`，这很好；但默认值本身仍然容易踩坑。

建议默认改为：

```ts
outDir: options.outDir ?? 'wc'
manifestFile: options.manifestFile ?? 'zeus.components.json'
customElementsFile: options.customElementsFile ?? 'custom-elements.json'
```

`examples/react-wrapper` 里现在还传了 `dist/wc`、`dist/react`，这个也应该改成 `wc`、`react`。

## 5. `component-analyzer` 和 `component-dts` 放在 `addons/`，和之前设计定位不一致

现在：

```txt
addons/component-analyzer
addons/component-dts
```

但这两个包不是 bundler 插件，也不是某个构建工具适配层，而是 **核心共享分析/类型生成能力**。之前设计里更适合放：

```txt
packages/component-analyzer
packages/component-dts
```

当前构建脚本确实能扫描 `packages` 和 `addons`，所以短期不会坏；但长期维护语义不清晰。`component-analyzer` 自身 package 定义也更像核心库，而不是 addon。

建议二选一：

```txt
方案 A：现在就移到 packages/component-analyzer、packages/component-dts
方案 B：保留 addons，但把文档和命名改成“all compiler ecosystem packages live in addons”
```

我更建议 **方案 A**。

## 6. React/Vue examples 直接 import 虚拟模块，不是最终用户体验

例如 `examples/react-wrapper/src/App.tsx` 现在直接：


这个适合测试插件虚拟模块，但不符合最终想要的使用方式：


当前 example 里的注释也说明它依赖 bundler plugin 注册虚拟模块。
