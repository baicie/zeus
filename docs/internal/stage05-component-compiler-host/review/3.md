结论：**当前分支不建议直接合并 main，也不建议马上发布 0.1.0。**

这版已经明显接近最终方向：主插件已切到 `plugins?: ZeusComponentPlugin[]`，`DtsMode = boolean | 'auto'`、默认 include/exclude、output registry、`output-css`、`preset-component-library` 都已经落地。  
但当前代码仍有几个 **P0 阻塞**，会导致 `pnpm check` / examples build / 发布包行为不稳定。

---

# 总体判断

```txt
是否符合架构方向：是
是否可以合 main：暂不建议
是否可以发布 0.1.0：不建议
建议状态：修完 P0 后再合 main；发布前再修 P1
```

# P0：合并 main 前必须修

## 1. 还有旧 API：`outputs` 没迁完

当前 `ZeusBundlerPluginOptions` 里已经没有 `outputs`，只剩 `plugins`。

但至少这些文件还在使用旧写法：

```ts
zeus({
  outputs: [wc(), react(), vue()],
})
```

例如 benchmark 里仍然是 `outputs`：

```ts
zeus({
  root: __dirname,
  components: {
    include: ['src/components/**/*.{ts,tsx}'],
  },
  outputs: [
    wc(...),
    react(...),
    vue(...),
  ],
})
```

React wrapper example 也还在用 `outputs`，并且还传了已经被移除的 `wcOutDir`。
Vue wrapper example 同样如此。
Web Component example 也还在使用 `outputs`。

这会直接导致：

```txt
1. TypeScript 配置检查失败
2. example build 失败
3. benchmark 失败
4. CI 结果不可信
```

尤其 `benchmarks/component-host/vite.config.ts` 被 root `tsconfig.json` include 进来了。
所以这是明确 P0。

**建议修复：**

```ts
zeus({
  plugins: [wc(), react(), vue()],
})
```

同时删除所有：

```ts
wcOutDir
outputs
root: __dirname // Vite 下可省略
```

---

## 2. `output-css` 在 ESM 包里使用了裸 `require`

`@zeus-js/output-css` 的 package 是 `"type": "module"`，并且 `import` 条件指向 ESM 产物。

但 `processCss.ts` 里直接使用了：

```ts
require('sass')
require('less')
require('postcss')
require('postcss-load-config')
require('lightningcss')
```

这在 ESM 产物里有运行时风险。默认 copy CSS 不一定触发，但只要用户用了 `sass/postcss/less/minify` 就可能出问题。

**建议修复：**

用动态 import 或 `createRequire(import.meta.url)`。更推荐动态 import：

```ts
const sass = await import('sass')
const less = await import('less')
const postcss = await import('postcss')
const loadConfig = await import('postcss-load-config')
const lightningcss = await import('lightningcss')
```

同时 `package.json` 需要补齐 peerDependencies。现在 `output-css` 只声明了 `lightningcss` 和 `rollup`，但源码还会加载 `sass / less / postcss / postcss-load-config`。

建议改成：

```json
"peerDependencies": {
  "postcss": "*",
  "postcss-load-config": "*",
  "sass": "*",
  "less": "*",
  "lightningcss": "*"
},
"peerDependenciesMeta": {
  "postcss": { "optional": true },
  "postcss-load-config": { "optional": true },
  "sass": { "optional": true },
  "less": { "optional": true },
  "lightningcss": { "optional": true }
}
```

---

## 3. 发布 0.1.0 的版本策略还不完整

当前根版本和核心包版本仍是 `0.0.2`。

发布脚本会根据 changeset fixed group 生成版本变更，但 `.changeset/config.json` 的 fixed group 只包含核心包：

```txt
@zeus-js/shared
@zeus-js/signal
@zeus-js/runtime-dom
@zeus-js/compiler
@zeus-js/zeus
```

而新的 compiler-host 相关包：

```txt
@zeus-js/bundler-plugin
@zeus-js/component-analyzer
@zeus-js/component-dts
@zeus-js/output-wc
@zeus-js/output-react-wrapper
@zeus-js/output-vue-wrapper
@zeus-js/output-css
@zeus-js/output-icons
@zeus-js/preset-component-library
```

不在 fixed group。发布脚本生成 changeset 时只给 `fixedGroupPackages` 写 bump。
但真正 publish 时又会发布所有非 private 且未 ignore 的 workspace 包。

这会导致一个很尴尬的结果：

```txt
核心包可能变成 0.1.0
web-c 新包可能仍是 0.0.2
@zeus-ui/registry / zeus-ui 可能以 0.0.1 被一起发布
```

`@zeus-ui/registry` 当前非 private 且版本是 `0.0.1`。
`zeus-ui` CLI 也是非 private 且版本是 `0.0.1`。

**发布 0.1.0 前必须明确策略：**

方案 A，所有主线包一起 0.1.0：

```json
"fixed": [
  [
    "@zeus-js/shared",
    "@zeus-js/signal",
    "@zeus-js/runtime-dom",
    "@zeus-js/compiler",
    "@zeus-js/zeus",
    "@zeus-js/bundler-plugin",
    "@zeus-js/component-analyzer",
    "@zeus-js/component-dts",
    "@zeus-js/output-wc",
    "@zeus-js/output-react-wrapper",
    "@zeus-js/output-vue-wrapper",
    "@zeus-js/output-css",
    "@zeus-js/output-icons",
    "@zeus-js/preset-component-library"
  ]
]
```

方案 B，只发布核心运行时 0.1.0，compiler-host 暂不发布：

```json
"ignore": [
  "@zeus-js/bundler-plugin",
  "@zeus-js/component-analyzer",
  "@zeus-js/component-dts",
  "@zeus-js/output-wc",
  "@zeus-js/output-react-wrapper",
  "@zeus-js/output-vue-wrapper",
  "@zeus-js/output-css",
  "@zeus-js/output-icons",
  "@zeus-js/preset-component-library",
  "@zeus-ui/registry",
  "zeus-ui"
]
```

如果目标是“component compiler host 作为 0.1.0 主卖点”，建议选 **方案 A**。
如果目标是“0.1.0 先稳核心框架”，建议选 **方案 B**。

# P1：发布前强烈建议修

## 4. PR CI 没跑完整发布前检查

`component-host-rc.yml` 的 fast job 只跑了：

```txt

pnpm check
pnpm test-unit
```

bench job 只跑了：


```txt

```

虽然 `release:precheck` 里有这些步骤。
但那是在打 tag 发布时才跑，太晚了。合 main 前就应该跑。

**建议新增 integration job：**

```yaml

- run: pnpm build-dts
- run: pnpm check:exports
- run: pnpm examples:check:all
- run: pnpm check:registry-cli
- run: pnpm docs:build
```

## 5. `packages/headless` workspace 项容易误导

`pnpm-workspace.yaml` 里仍然有：

```yaml
- 'packages/headless'
```

但你现在已经明确 `@zeus-ui/headless` 是实战 fixture，实际位于 `examples/headless`，并且是 private。

这不是功能阻塞，但容易让后续维护者误解。

```yaml
- 'examples/*'
```

当前 `output-css` 是新引入的发布包，并且承担替换手写 CSS emit 插件的职责。它已经有自动输入检测、processor 检测、CSS emit、watch 逻辑。


# 已经符合预期的部分

`ZeusComponentPlugin` 已经有：

```txt
setup
buildStart
virtualModules
generateBundle
external
```

主插件也按顺序调用了：

这是符合最终设计的。

`resolveDts()` 已经支持：

```txt
显式 true / false
package types 字段
typescript dependency
tsconfig.json
.ts/.tsx source
```

插件级 `resolvePluginDts()` 也已经实现了“插件显式配置优先，否则跟随 ctx.dts.enabled”。

`createOutputRegistry()` 已经支持：

```txt
register
has
get

getFileName
join
```

并且内置了 `wc/react/vue/icons-*` 默认目录。

Vue wrapper 同样如此。

Vue prop sync 也已经改成无条件同步：

```ts
el.xxx = props.xxx
```

这符合你前面定的最终方向。

`@zeus-ui/headless` 仍是 private，并放在 `examples/headless`。

按你当前定位，它不是主线发布包，而是接近真实组件库的验证场景，所以这个可以保留。
